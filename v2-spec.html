把“设备级 agent”收敛成一套可实现的主权执行平面。

这份文档记录当时收敛出的 v1 参考实现思路，保留较强实现偏好，供今天回看取舍。核心原则只有一句： 智能可以变化，权限边界必须稳定。 因此当时的实现判断是规则优先、本地常驻、票据驱动、分级隔离和可回滚执行。

System Boundary

设计边界先固定，再谈智能

全权操作设备不是“一个模型拿到最大权限”，而是系统在边界内覆盖感知、规划、执行、约束和回滚。下面所有模块都服从这个前提。

North Star

Authority lives in the kernel

真正的控制权只属于本地守护内核。planner 只能提出计划和受限工具请求，不能直接拥有主机权限。

Power Baseline

CPU-only 必须可运行

低功耗基线是事件路由器、缓存、宏和规则引擎，不是本地小模型。模型只是可选增强层。

Recovery First

Every write needs a way back

任何写操作都必须绑定 postcondition 和补偿路径。无法回滚的动作先草稿、再确认、最后提交。

Historical Reference Implementation

v1 参考实现思路

这部分记录当时偏好的实现路径，供当前规范回看和取舍，不再是默认实现要求。

Host Core

Rust + Tokio daemon

负责事件订阅、任务状态机、票据发放、执行编排、审计和回滚。优点是低常驻开销、长时间运行稳定、并发可控。

State + Policy

SQLite WAL + Cedar policies

状态索引、记忆索引、宏、工件索引和审计链落到 SQLite；每个任务再保留 append-only JSONL 时间线用于重放、导出和旋转归档。策略用 Cedar 表达 scope、resource、approval 和风险条件。

Local Reflex

Rules-first router, optional small model

默认由规则、缓存、宏和风险矩阵常驻；有 GPU/NPU 时，再接小模型做分类、摘要和状态压缩。

Cloud Planner

OpenAI Responses API + shell/computer

只处理复杂、陌生、跨应用长链任务。返回计划、校验标准和有边界的工具请求，不直接控制本机。

Config + Diagnostics

User / project / local hierarchy + doctor

配置分为全局、项目、本地临时覆盖三层，运行时只消费合并后的 typed config。内建 doctor、status、sandbox 诊断面，不把故障排查留给猜测。

Parity Harness

Deterministic mock provider + executor harness

任何 provider、执行器、审批和回滚路径都先走可重放的 mock parity harness，再进入真实设备。这样验证的是系统契约，而不是一次偶然成功。

Module Ownership

模块边界

planner、broker、executor 三者不能共用主权。每个模块只负责一层因果，不跨权操作。

Event Ingest
window / file / timer / notification
把原始系统事件标准化成 EventEnvelope，去噪、聚合、附上来源和时间戳，不做权限判断。
失败模式：噪声过高。对策：去重、节流、优先级队列。
Reflex Router
surface + mode chooser
根据缓存、宏、风险、硬件和可用表面，决定走 API、UIA/DOM、浏览器还是像素，并决定 direct/script/macro/L2。
失败模式：误升级昂贵路径。对策：强制 surface order 和 budget policy。
Capability Broker
policy + approval gate
依据策略、风险等级和审批结果发放短期 CapabilityTicket，并写入审计链。
失败模式：策略绕过。对策：所有 executor 先验票再执行。
Plugin + Skill Manager
manifest + lifecycle
插件和技能通过显式 manifest 注册能力、hook、生命周期和所需权限。扩展可以存在，但扩展不能越过主权边界。
失败模式：扩展失控。对策：manifest 权限声明 + 独立健康检查 + 可禁用。
Degraded Lifecycle Monitor
partial availability
对 browser、MCP、plugin、VM executor 的启动、握手、发现、调用和关闭阶段做生命周期跟踪，输出 degraded report，而不是全局 hard fail。
失败模式：一个子系统挂掉拖垮全局。对策：phase-based health + available/missing capability report。
Recovery Coordinator
recipe registry
把已知故障映射成 recovery recipe，例如重试握手、重建 worker、重新绑定表面、请求人工确认。默认只自动恢复一次，避免死循环。
失败模式：无穷重试。对策：single auto-recovery budget + escalation policy。
Lifecycle

统一状态机

稳定性来自状态机，而不是提示词。任何模块都只能推动状态前进，不能绕过状态机直接做事。

1. observed事件进入队列，附带来源和时序。
2. normalized去噪、聚合、标准化。
3. triaged计算风险、预算、硬件档位、缓存/宏命中。
4. planned形成 direct/script/macro plan 或 L2 draft。
5. approvedbroker 验策略，必要时获取用户审批，签出 ticket。
6. executing执行器持 ticket 逐步动作，写 receipt。
7. verifying按 postcondition 重读状态，决定成功或失败。
8. completed落审计、更新宏、更新摘要。
9. failed / rolled_back失败入 quarantine，必要时补偿或回滚。
审批只发生在 planned -> approved，一旦 ticket 过期，状态必须退回 triaged 或 failed。
IPC Contracts

本地 IPC 契约

单设备 Windows 场景下，本地 RPC 的目标不是横向扩展，而是低复杂度、低暴露面和可调试。

Boundary Transport Payload Why This Wins
UI / local client -> core daemon Windows named pipe versioned JSON message envelope 仅本机可见，无需开放端口，权限边界和调试都足够清晰。
Core daemon -> child worker stdin/stdout JSON lines ExecutionStep / CapabilityTicket / ExecutionReceipt 适合 Playwright、PowerShell、Python worker 这种短命子进程，没有额外服务发现成本。
Core daemon -> isolated browser named pipe + artifact folder control JSON + traces / screenshots / DOM snapshots 控制流和大体积工件分离，方便重放、裁剪和审计。
Core daemon -> disposable VM worker ephemeral pipe bridge scoped commands + short-lived tokens VM 生命周期短，不值得引入长期 agent 协议。
Core daemon -> cloud planner HTTPS scrubbed context + bounded tool manifest 天然适合 provider 边界，也便于和本地主权层隔离。

Capability Ticket

能力票据

任务不是最小权限单位，步骤才是。每个可执行步骤都必须绑定 ticket。

Ticket fields
  • scope：例如 filesystem.rename、browser.click。
  • resource_bound：路径、窗口、域名、账户、进程等资源约束。
  • ttl_sec：默认 60 到 300 秒，超时即失效。
  • sandbox_tier：host-limited、browser-isolated、vm-ephemeral。
  • approval_ref：关联用户确认、策略白名单或批次预授权。
  • policy_context：记录 rule/hook/health gate 对本次动作的附加限制。
  • audit_id：全链路审计和回滚锚点。

Issue

Broker signs ticket

planner 不能签票，executor 也不能自签。只有 broker 在策略和审批都满足时发票。

Verify

Executor validates locally

执行器在每一步开始前本地验签、验 TTL、验资源范围，任何一项不符立即拒绝。

Expire

Ticket expiry is a hard stop

票据过期后不允许自动续签。系统必须回到 triage 或 human gate，避免静默权限升级。

Isolation Topology

隔离拓扑

隔离是分级释放，不是二元切换。不是所有风险都直接上 VM。

Isolation ladder
  1. host-limited：普通读操作、低风险写操作，宿主最小权限。
  2. browser-isolated：陌生网页、站点登录、跨域自动化，独立浏览器 profile 与下载目录。
  3. app-container：来源不明但无需整机环境的桌面应用或脚本。
  4. vm-ephemeral：安装包、输入凭据、远程连接、高风险第三方集成。
Do not do this

两个常见错误

一是所有动作都在宿主跑，最后靠“模型谨慎”补安全；二是凡是风险就全进 VM，最后连稳定自动化都变得过慢过重。

正确做法是用票据限制“能做什么”，再用隔离降低“做错损失有多大”。

Isolation routing rule 
if action in HIGH_RISK or input_credentials or install_package:
    sandbox = "vm-ephemeral"
elif browser_surface and site_trust != "known-good":
    sandbox = "browser-isolated"
elif desktop_binary and publisher_unknown:
    sandbox = "app-container"
else:
    sandbox = "host-limited"
Power + Scheduling

调度与功耗

真正决定功耗的是事件密度、截图频率、云调用时机和重复流程是否被宏化。

Scheduler policy 
loop:
  evt = next_event()
  env = normalize(evt)
  if cache_hit(env):
      dispatch(cache_plan)
      continue
  if macro_hit(env):
      dispatch(macro_plan)
      continue
  if deterministic_surface_exists(env):
      dispatch(script_or_direct_plan)
      continue
  if risk_high(env) and no_approval(env):
      park_for_human()
      continue
  if budget_allows_cloud(env):
      dispatch(l2_plan_request)
  else:
      park_or_degrade()
CPU-only tier 常驻只有事件监听、规则引擎、缓存、宏和 SQLite。无持续截图，无常驻云连线。
GPU/NPU tier 小模型只处理摘要、分类、候选计划重排和轻量问答，不接管执行主权。
Cloud budget 每个任务显式声明最大云调用次数、最大截图数、最大运行时长，超出则降级或暂停。
Macro promotion 连续成功且无人工修正的重复流程才允许升格为宏，并必须带回归样例。
Long-Term Memory

长期记忆必须是类型化、受控、可撤销的

设备级 agent 不能把“长期记忆”理解成无限聊天记录。真正可用的长期记忆应该是 按类型分层、按价值晋升、按作用域检索、按证据失效。 这样记住的是稳定事实、偏好、流程和故障经验，而不是把全部上下文永久囤积。

Five-layer memory model
  • Policy memory：长期规则、审批偏好、禁区、默认预算、风险偏好。
  • Semantic memory：设备、应用、账户、目录、网站、联系人、常用实体之间的稳定关系。
  • Procedural memory：成功 workflow macro、恢复配方、已验证执行路径。
  • Episodic memory：最近任务摘要、故障经过、人工纠正记录、异常事件。
  • Trust memory：站点/应用/目录/插件/执行器的信任级、健康级和历史稳定性。
Not raw transcript

什么不应进入长期记忆

  • 原始全量对话和整屏截图不直接升格为长期记忆。
  • 密钥、密码、完整 token、原始 cookie、未脱敏凭据不写入记忆层。
  • 一次性上下文、未验证猜测、单次失败后的幻觉结论不应长期保留。
  • 高敏感内容只保留句柄、摘要或 provenance，不保留明文。

Promotion pipeline 
candidate -> validate -> promote -> refresh -> expire/revoke

if repeated and verified and low_contradiction:
    promote(memory)

    lower_confidence(memory)

    archive(memory)
Write Gate

不是每次交互都记住

只有满足重复、价值高、被验证、被用户确认或持续稳定的内容，才允许从 episodic 提升到 semantic / procedural。

Retrieval

按任务阶段取不同记忆

规划前取 policy + semantic，执行前取 procedural + trust，失败后取 episodic + recovery。避免全量召回污染当前任务。

Forgetting

遗忘是功能，不是 bug

Episodic 默认 TTL，semantic/procedural 需要定期再验证；一旦环境变化或被反证，记忆必须降权、失效或撤销。

记忆层和会话层要分开：sessionlog 负责完整可回放，memory 负责长期可检索。前者保存经过，后者保存结论，而且结论必须带证据和失效条件。
Reliability

失败处理

设备级 agent 最贵的不是一次失败，而是“做了一半、状态未知、权限还没收回”。

Failure type Primary response Required artifact Escalation
Ticket expired mid-flight 立即停止下一步动作，标记当前步为 interrupted step receipt + current state snapshot 返回 triaged 或要求重新审批
Postcondition timeout 执行独立重读，不重复写操作 read-after-write evidence 进入 failed，再决定补偿或人工介入
Executor crash 回收票据，终止相关 worker，保留工件 stdout tail, trace, last heartbeat 同类动作进入 quarantine
Planner proposes out-of-scope step broker 直接拒绝，不进入执行 rejected plan draft 降级为 deterministic path 或人工确认
Rollback failed 冻结相关资源，阻止后续自动动作 compensation receipt + diff 强制人工接管
所有失败最终都要落到三种结局之一：安全回退、人工接管、隔离重试。不能存在“不知道现在是什么状态”的悬空态。默认策略是 一次自动恢复 + 明确升级，而不是无限重试。
Refinements From Claw Code

从公开 rewrite 里抽出来的可借鉴工程点

这里借鉴的不是它的产品定位，而是它在工程组织上的几个成熟模式。它更像 coding harness；我们只吸收对设备代理同样成立的结构化经验。

Borrow

应该吸收的部分

  • 把运行时、provider、tools、plugins、commands、telemetry、compat-harness 分开，不把所有逻辑塞进一个主二进制。
  • 把 degraded mode 作为一等公民处理，允许部分能力缺席但整体继续运行。
  • 保留 append-only 时间线，便于重放、导出、事故排查和会话恢复。
  • 维护显式 recovery recipe，而不是把故障恢复留给隐式 prompt 技巧。
  • 内建 doctor / status / sandbox 诊断面，缩短运维回路。
  • 先做 mock parity harness，再碰真实 provider 和真实设备。
Do not import blindly

不应该照搬的部分

  • 不照搬 coding-agent 的命令面膨胀。设备代理的核心界面应该更少、更稳。
  • 不照搬默认高权限。设备代理必须坚持 capability ticket，而不是以 `danger-full-access` 为常态。
  • 不把工作区级权限模型直接当成设备级权限模型。文件写权限不等于设备动作授权。
  • 不把“会话恢复”误当成“系统恢复”。设备代理还需要表面绑定、执行器健康和回滚补偿。
Landscape Comparison

和 Claw Code、OpenClaw、Claude Code、Codex、OpenCode 的关系

这套设计不是另一个 coding agent，而是设备级主权控制层。下面会直接说明差异和优势。

Positioning

它解决的问题层级更低

OpenClaw 偏个人助理入口，Claude Code / Codex / OpenCode 偏编码 agent 或 agent harness；这套设计偏“设备级自治执行平面”。

也就是说，它不是要替代这些工具的全部体验，而是给“长期自治 + 整机控制”补上它们通常不作为第一优先级去解决的那层结构。

Core advantage

主权边界比模型能力更靠前

这里优先设计的是 capability ticket、统一状态机、分级隔离、低功耗调度、可恢复失败，而不是先追求能跑通更多 demo。

System Primary center of gravity What it is best at Where this design is stronger
Claw Code 公开 Rust rewrite 的 coding harness crate 拆分清晰，provider/runtime/tools/plugins/parity-harness 工程组织成熟 设备主权边界、逐步授权、分级隔离、真实设备恢复策略比它更强；但我们已吸收它在 degraded mode、诊断面、append-only timeline、mock parity 上的长处
OpenClaw 个人助理 / 网关型 agent 常驻 daemon、多渠道入口、多模型整合、更像现成产品 设备主权边界、逐步验票、统一回滚状态机、分级隔离和低功耗调度更系统化
Claude Code 开发工作台 agent 终端、IDE、桌面、浏览器里的编码与开发者工作流 跨应用设备控制、宿主主权、执行分层、风险治理和长期自治约束
Codex 云端并行编码代理 仓库级软件工程、云端后台任务、GitHub/PR、并行委托 本地事件驱动、整机动作治理、低功耗常驻、宿主侧安全边界
OpenCode 本地可配置 agent harness rules、permissions、subagents、skills、本地 CLI 工作流 完整设备代理架构，明确规定 broker、ticket、verifier、sandbox、rollback 的因果关系
一句话：它们大多优化任务执行体验，我这套方案优先优化主权边界和长期自治成本。

Acceptance Gates

验收门槛

“看起来能跑”不算完成。可交付必须量化到功耗、安全、恢复性和成本下降趋势。

Power

空闲不做多余事

CPU-only 档位下空闲态不能持续截图，不能持续云调用，核心服务应维持低占用并可长时间运行。

Safety

所有执行先验票

抽样检查任意动作，必须能追到 ticket、approval_ref、artifact、receipt 和 verifier 结果。

Reliability

失败后状态可知

任何失败都必须停在 completed、rolled_back、paused_for_human 三者之一，不能留下未知状态。

Coverage

结构化路径优先达成

固定任务集里，绝大多数稳定流程应优先命中 API、UIA/DOM 或 deterministic script，而不是像素兜底。

Economics

重复任务越来越便宜

宏命中率随时间上升，frontier token、截图次数和人工确认率随时间下降。

Recoverability

高风险有补偿路径

删改、安装、外发、改权限等动作必须具备草稿、快照、撤回、undo manifest 或双阶段提交之一。

Parity

Mock harness 先于真机验证

provider、执行器、审批、回滚和 degraded mode 都必须先通过 deterministic parity harness，再进入真实设备环境。

Memory

长期记忆可查、可删、可失效

任意长期记忆都必须能追到来源、置信度、最近验证时间和撤销条件；用户必须可查看、修正和删除。

这些门槛的目的不是追求完美，而是避免系统在“看起来很智能”的同时，实际上不可控、不可审计、不可恢复。
