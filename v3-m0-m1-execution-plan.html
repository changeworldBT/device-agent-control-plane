1. Scope Lock

这一轮只允许做什么

In
  • Replay A 的完整状态推进
  • append-only event log
  • materialized state view
  • Task / TaskNode / Fact / Grant / Receipt / Verification 的最小运行时
  • Selector v0：硬约束 + 默认评分
  • Prepare 节点的可验证完成
Out
  • Commit 节点真实执行
  • compensation 执行链
  • 多 agent orchestration
  • 外部 workflow / memory / agent adapter
  • pixel fallback 和 GUI automation
  • procedure trusted 自动命中

Gate: 如果团队发现必须先做上面 Out 里的东西才能推进 M0/M1，优先级判断就是错的，应回退并重新切片，而不是扩大范围。

3. Work Packages

M0 / M1 的最小工作包

W1

Schema Ingestion and Replay Loading

  • 直接消费 schemas/json-schema 和 schemas/examples
  • 把 replay fixture 读成可执行 timeline
  • 将 validation failure 变成阻塞错误，而不是运行时默认补字段

Done: Replay A fixture 能在无业务代码的情况下完成结构校验和 timeline 装载。

W2

Event Log Spine

  • 稳定 event id、causation、correlation、trace
  • append-only write
  • dedup by event_id
  • replay from log

Done: 删除 materialized view 后，仍能从事件流重建当前状态。

W3

Materialized State View

  • TaskNode current state
  • Fact current state + stale downgrade
  • Grant active set
  • Receipt / Verification lookup

Done: projector 输出与 replay 期望一致，且 conflict facts 可并存。

W4

Task and Fact Runtime

  • TaskNode transition guards
  • Fact time fields: observed_at / verified_at / valid_until / ttl
  • Fact conflict_set / supersedes
  • candidate -> verified promotion only through verification

Done: 过期或冲突事实不会静默覆盖当前视图。

W5

Grant, Execution, Verification Vertical Slice

  • grant issuance for low-risk structured actions
  • execution dispatch only with active grant
  • receipt recording
  • verification-backed terminal eligibility

Done: 无 grant dispatch 被拒绝；无 verification terminal transition 被拒绝。

W6

Selector v0 and Attention Budget

  • hard constraints: grantable / verifiable / within policy / within budget
  • default scoring: freshness / health / verification strength / risk / user attention / cost
  • attention budget enters selection outcome

Done: Replay A 能在不引入无意义 ask 的情况下完成 Prepare terminal。

W7

Minimal Capsule Export

  • 只实现 resume / handoff 最小导出
  • hard versus soft constraints must be explicit
  • capsule validator only checks M0/M1 required fields

Done: 研究简报任务可在中途生成可校验 capsule，但 capsule 不是推进终态的必要条件。

6. Stop Conditions

出现这些情况，不要继续加功能

Stop

如果状态视图不能从事件流重建，说明真相面实现方向错了。

Stop

如果事实还是被当成无时间静态值，说明 Fact 语义没有真正落地。

Stop

如果 grant / verification 还能被绕开，说明系统仍然只是工具编排。

Stop

如果 Replay A 需要频繁 ask 才能完成，说明 attention budget 没进 selector。

Stop

如果团队开始为了“以后会用到”而引入多 agent 或外部兼容层，说明 backlog 失焦。

Stop

如果 capsule 被当作状态权威副本使用，说明主状态面和导出资产已经混了。

7. Exit Gate

M0 / M1 通过后，才允许进入下一阶段

Must Be True
  • Replay A 全链路跑通
  • 事件流 -> 物化视图重建稳定
  • Fact 时间 / 冲突语义有测试覆盖
  • grant / verification guard 都有效
  • Prepare terminal 可验证，不被误当 Commit
Only Then

才进入 M2 Controlled Operate / Commit Slice，也就是 CRM 写入、R2/R3 approval、rollback / compensate 分流、以及 Replay B 的真实执行闭环。
