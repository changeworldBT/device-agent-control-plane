April 8, 2026 · Current agent reality, engineered response

在当前约束下，更好的 agent 不是更大权限，而是更小的可验证内核。

目标不是让一个前沿模型直接接管整台设备，而是构建一个 分层主权代理蓝图： L0 安全内核负责边界与回滚， L1 本地低功耗反射代理负责常驻监听与简单动作， L2 云端深度规划代理只在复杂任务时被按需唤醒。 在当前 agent 能力还不够稳定的现实下，这种分层更容易同时满足效率、安全、低功耗与覆盖面；正式冻结规范以 v3-design-freeze.html 为准。

现状判断

2026 年的 computer-use agent 已经能在网页、终端、表格和轻量 GUI 工作流里高效完成任务，但对整机、长链、无监督 OS 级控制仍不够稳。裸放权只会放大误触、权限失控和能耗浪费。

设计结论

把“思考”与“执行权”拆开。模型提出计划，系统内核发放短期能力票据，执行器只拿到完成当前动作所需的最小权限，且所有高风险动作必须可回滚、可审计、可中止。

当前技术判断

当前偏好的运行方式是：本地常驻基线先由事件路由器和规则引擎承担；有合适算力时再叠加小模型做分类与压缩。云端 frontier model 只处理复杂规划，所有结构化接口优先于视觉操作；重复任务逐步编译为确定性宏，减少后续推理成本与功耗。

First-Principles North Star

设计原则

从第一性原理拆解，“全权操作设备”不等于“无限权限”。它本质上是四件事： 感知、规划、执行、约束。当前行业现状更支持异构分层，而不是单体全能模型。

01

结构化优先

能调 API 就不看屏幕，能读 Accessibility/DOM 就不做视觉推理，能跑宏就不让模型重想一遍。这样最稳、最省电、最可验证。

02

权限按动作发放

没有长期 root。只有短期、最小化、带用途说明和 TTL 的能力票据。模型永远不直接持有设备主权。

03

事件驱动常驻

常驻代理不该不停截图轮询，而应监听窗口变化、通知、前台切换、定时器和任务队列，在真正有事时才唤醒推理。

04

高风险动作稀疏确认

不是每步都确认，而是只在支付、删除、安装、改权限、外发内容、输入敏感信息等高风险边界确认。

05

经验必须编译

成功重复 3 次的流程不应继续消耗大模型，应沉淀成可重放、可回归测试、可版本化的 workflow macro。

System View

三层主权架构

这是整套方案的核心。L0 负责边界，L1 负责低功耗常驻，L2 负责复杂推理。 设备“全权可操作”来自整体覆盖面，而不是把无限权限交给任何一个模型。

Execution Strategy

动作执行优先级

设备 agent 的成功率、功耗和安全性，主要取决于两个决策先后顺序是否正确： 先选交互表面，再选执行方式。下面这条梯子只描述“表面优先级”，不是把脚本、宏和模型混成一条线。

R1
API / Structured RPC 直接调用系统 API、应用 API、数据库或服务接口。最稳、最省电、最可回放。
default path
R2
Accessibility / DOM Windows UIA、Web DOM、app semantic tree。对 GUI 有理解，但仍保持结构化控制。
preferred ui path
R3
Browser Automation 对 Web 业务使用 Playwright 等工具，只有在 API/DOM 缺失时才进入浏览器控制层。
fallback
R4
Pixel Computer Use 鼠标键盘截图级控制。只用于未知桌面、远程窗口、视觉密集 app 或最后一公里兜底。
last resort

为什么这条梯子更稳

  • 结构化路径的验证成本最低，可以直接做 schema 校验、状态检查和幂等重试。
  • 像素级控制最消耗 token、截图和重试时间，也最容易被 UI 变化击穿。
  • 对重复任务，宏编译会把“会做”转化成“低成本稳定做”，但宏属于执行方式，不改变表面优先级。
  • 这条顺序天然支持风险治理：越往下越接近人工桌面，也越需要强审计和确认。
Two-Stage Routing Rule 
surface = choose_surface(
    api,
    accessibility_or_dom,
    browser_automation,
    pixel_computer_use
)

mode = choose_mode(
    direct_call,
    deterministic_script,
    compiled_macro,
    l2_planner
)

run(surface, mode, capability_ticket)

执行方式的默认顺序是：direct call > deterministic script > compiled macro > L2 planner。 也就是先决定“在哪个表面操作”，再决定“用多智能还是多确定性的方式去操作”。

Runtime Loop

事件驱动任务流

更可取的方案不是“每秒截图 + 一直问模型下一步”。而是把系统做成反应式流水线： 有事件才更新世界状态，有不确定性才调用深度规划，有高风险才要求确认。

Technical Direction

当前技术判断

这一节记录当前偏好的实现读法，不替代正式冻结规范。判断优先级仍然是： 可靠边界、功耗效率、结构化控制、可替换模型后端，最后才是开发便利。

Core Daemon
Rust + Tokio service
负责事件订阅、任务调度、票据发放、审计、快照、执行器编排。Rust 是因为需要低常驻开销、内存安全和长时间稳定运行。
Windows 首选宿主。单机守护进程，不引入重型分布式组件。
Policy Engine
Cedar policy + local capability broker
用显式策略表达谁可以在什么上下文里执行什么动作，并为每次执行发放短期 capability ticket。
避免“模型拿永久权限”的设计错误。
Secrets
Windows DPAPI / Credential Manager
密钥永远不进入模型上下文，只在执行器侧按需取用；对外系统凭据通过代理令牌或短期交换票据获取。
降低 prompt injection 和日志泄露面。
State Store
SQLite + WAL
保存任务队列、事件索引、状态摘要、能力票据、宏版本、回滚点、操作日志。足够快，部署简单，功耗低。
单机系统不需要 Kafka、Temporal、Postgres 组合拳。
L1 Model
Rules-first event router + optional small local model
默认由规则引擎、缓存和宏匹配常驻；在具备合适 GPU/NPU 时，再叠加 Phi-4-mini 或 Gemma 3n 级模型做分类、摘要、风险预判和状态压缩。
CPU-only 也可运行；有加速器时优先 ONNX Runtime + DirectML。
L2 Planner
V1 primary: OpenAI Responses API + shell/computer
负责复杂规划、长链恢复、未知 UI、跨系统数据整合。其他 providers 作为兼容后端，不放进 v1 默认控制路径。
云端只在必要时唤醒，降低成本与能耗。
UI Executor
Windows UIA adapter + DOM adapter + Playwright
把 GUI 和浏览器的“可见界面”尽可能转成结构化动作。浏览器流程优先 Playwright worker，不让模型手点网页。
像素级 computer use 只做最后兜底。
Sandbox
Host-limited runner + isolated browser/app container + disposable VM
普通动作在宿主最小权限下执行；不可信网页先进入隔离浏览器或低完整性/AppContainer；安装包、跨站登录、输入凭据和高风险自动化再进入一次性 VM/容器。
把“设备全权”拆成“宿主安全 + 分级隔离释放能力”。
Observability
Structured event log + replay timeline
所有动作都记录 precondition、ticket、tool call、postcondition、artifact 和 rollback pointer。
调试和追责都靠重放，不靠猜。

核心接口契约

让 planner、broker、executor 解耦，关键不是 prompt，而是边界对象必须稳定。 下面这 4 个对象建议作为系统最小契约。

  1. TaskSpec：用户目标、约束、风险等级、成功定义。
  2. ExecutionPlan：分步动作、依赖、预期状态、校验方法。
  3. CapabilityTicket：动作范围、资源范围、TTL、审计 ID。
  4. VerificationResult：是否达成、证据、差异、是否回滚。
Minimal Control Contract

  "task_spec": {
    "goal": "book meeting room and send invite",
    "constraints": ["no external share", "work hours only"],
    "risk": "medium",
    "success": ["calendar entry exists", "invite delivered"]
  },
  "capability_ticket": {
    "scope": ["calendar.write", "mail.send_internal"],
    "ttl_sec": 180,
    "sandbox": "host-limited",
    "audit_id": "evt_20260408_001"
  }

Security + Power

安全与低功耗治理

设备级代理的成败不在“能不能做”，而在“做错时会不会很贵”。所以安全和功耗不是附属模块，而是主设计轴。

功耗设计目标

  • Idle watch mode：常驻不轮询，目标是接近零云调用和接近零持续截图。
  • Wake on event：由窗口变化、通知、任务队列、文件变化和定时器触发，不做盲扫。
  • State compression：L1 只维护摘要和最近窗口状态，不缓存整屏历史。
  • Burst compute：复杂任务短时唤起 L2，完成后立即释放，不让重推理常驻。
  • Hardware tiers：CPU-only 设备走规则优先；GPU/NPU 设备再叠加本地小模型。
  • Macro dominance：成熟期目标是多数重复任务不再调用 frontier model。

安全设计目标

  • Zero standing privilege：无永久管理员权限，无永久登录态暴露给模型。
  • Action-level approval：默认按动作确认；同策略批次只允许有限预授权，且仍逐步校验。
  • Reversible by default：删除、写入、安装、外发优先走有快照/草稿/暂存的路径。
  • Sandbox for uncertainty：陌生网页、下载、第三方安装包按不确定性进入分级隔离环境。
  • Full replayability：任何自动动作都能回看输入、票据、输出和验证结果。

风险矩阵

为什么这比“单前沿模型 + 最大权限”更优

  • 更高成功率：大部分稳定动作被下沉到确定性路径，不被 UI 抖动反复打断。
  • 更低功耗：本地小模型常驻，云端推理只在必要时发生。
  • 更安全：高风险动作被票据、审计、快照和确认包住，而不是靠模型“自己谨慎”。
  • 更可持续：每次成功执行都能沉淀成宏，系统会随使用变得更便宜，而不是一直昂贵。

明确放弃的设计

  • 放弃“一个模型拥有永久设备主权”。
  • 放弃“持续截图 + 持续推理”的高功耗常驻模式。
  • 放弃“所有 GUI 都用像素控制”的低效路径。
  • 放弃重型分布式编排栈在单设备上的过度工程化。
  • 放弃把 secrets 和系统长期状态直接暴露给 planner。
Landscape Comparison

和 OpenClaw、Claude Code、Codex、OpenCode 的区别

这套方案不是另一个“编码 agent 产品”，也不是简单的“电脑代操作工具”。 它的定位是设备级主权控制层：先固定权限、状态机、隔离和回滚，再把不同模型或执行器接进来。

本方案的核心定位

  • 目标对象是整台设备和跨应用流程，不只是代码仓库或聊天入口。
  • 默认主语不是模型，而是本地安全内核、能力票据和执行状态机。
  • 模型后端是可替换部件，不能直接拥有长期权限。
  • 重点优化的是长期自治下的安全、功耗、回滚和审计，而不是只追求“能做出来”。

为什么这不是现成 harness 的重复发明

  • 现有产品大多以编码、终端协作、聊天入口或云端任务为中心，不以设备主权边界为中心。
  • 它们通常有很强的任务执行能力，但不一定内建“逐步验票、分级隔离、统一回滚状态机”。
  • 本方案把“低功耗常驻”和“可恢复失败”当成主设计轴，而不是附加功能。
OpenClaw
更像个人助理 / 网关型 agent
强项在于多平台、常驻助手、模型与渠道整合，离“个人 AI 助理产品”更近；它和本方案最接近，但主轴仍偏助手入口与集成能力，而不是设备主权内核。
本方案优势：逐步能力票据、统一状态机、分级隔离、回滚优先、低功耗调度更系统化。
OpenClaw 优势：上手更像现成产品，渠道/模型整合更直接。
Claude Code
更像编码工作台 agent
强项在终端、IDE、桌面和浏览器中的开发工作流，围绕代码理解、编辑、命令执行、MCP 和开发者协作展开。
本方案优势：把跨应用设备操作、安全边界、执行器分层和功耗治理放到核心。
Claude Code 优势：编码体验成熟、开发者工作流贴合度更高。
Codex
更像云端并行编码代理
强项在云端环境、后台并行任务、GitHub/PR 集成、代码仓库操作和受控互联网访问，适合把软件工程工作委托到云里跑。
本方案优势：宿主主权更强、本地事件驱动更低功耗、对整机和跨应用自动化更直接。
Codex 优势：云端并行工程任务、仓库级协作和背景执行更强。
OpenCode
更像可配置本地 agent harness
强项在规则、权限、agent/skills 组合、CLI 工作流和兼容 Claude Code 约定，适合作为本地开发和自动化框架。
本方案优势：不是单纯 harness，而是完整的设备代理参考架构，明确规定 broker、ticket、verifier、sandbox 和 rollback。
OpenCode 优势：灵活、开源、可定制，适合快速搭本地 agent 工作台。

一句话总结

OpenClaw 更像个人助理产品，Claude Code / Codex / OpenCode 更像编码 agent 或 agent harness， 而这套设计更像“设备级自治操作系统层”。 它不是替代它们所有能力，而是解决它们普遍没有作为第一优先级去解决的那部分问题： 主权边界、逐步授权、分级隔离、低功耗常驻、失败可恢复。

Implementation Path

五阶段落地路线

先建立边界，再建立执行，再引入高阶智能。顺序不能反，否则系统只会越来越危险而不是越来越强。

0

内核先行

先做 Rust daemon、SQLite 审计、能力票据、风险矩阵、快照和回滚，不接任何前沿模型。

Exit gate: 所有动作必须先拿票据；审计日志完整；回滚在受控样例上成功率 100%。

1

确定性工具层

接 UIA、DOM、Playwright、Shell、系统 API，先让 80% 稳定动作完全不依赖大模型。

Exit gate: 在固定任务集上，结构化路径成功率至少 95%，且无高风险越权动作。

2

本地反射代理

在具备合适算力时加入小模型做事件分类、状态压缩、风险预判、宏匹配和轻量问答；弱硬件保持 rules-first 常驻。

Exit gate: 空闲态无持续截图；CPU-only 档位可运行；有加速器时云调用频率明显下降。

3

云端深度规划

接入 frontier planner，仅处理复杂和陌生任务，并通过能力票据与执行层解耦。

Exit gate: planner 只能输出计划和受限工具请求；复杂任务失败后可中止、可回滚、可重放。

4

经验编译

把成功路径沉淀为 workflow macro 和 policy templates，逐步降低 token、截图和人工确认频率。

Exit gate: 重复任务的宏命中率持续上升，frontier token、截图次数和人工确认率持续下降。
