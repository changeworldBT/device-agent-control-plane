Truth Source

1. Problem Statement

系统要解决的不是聊天质量，而是可验证的状态变化

System must solve
  • 把高价值数字任务持续转化为可验证、可恢复、可复用的执行流程。
  • 在本地设备、网页、消息、文档和外部服务之间稳定推进跨表面任务。
  • 把重复成功和失败经验编译成长期资产，而不是积累原始聊天历史。
System does not solve
  • 不追求开放式闲聊体验最大化。
  • 不追求默认替代所有现有生态。
  • 不追求把所有重能力塞进基础安装包。
  • 不追求无边界全自动化。
2. Design Axioms

后续实现不得违背的冻结公理

Layer 0 · Immutable Axioms

任务比对话更基础；共享状态比上下文窗口更基础；证据比解释更基础；终态只能经验证或人工推进；高风险动作必须显式授权；经验必须经验证才能自动复用。

Layer 1 · Default Protocols

Fact / Artifact / Grant / Receipt / Capsule 的默认结构，Verify / Recover 的默认链路，多 agent 的默认协调原语，都是当前版本的默认协议。

Layer 2 · Adjustable Policies

任务分类、路径偏好、晋升阈值、capsule 形态、spawn 条件、TTL 和 freshness 策略都属于可调策略；调整这些不应被视为“改架构”。

任何设计元素都必须能回答两个问题：它解决哪个真实任务问题；删掉它会破坏哪个硬约束。固定的 6 类任务、路径顺序、procedure 晋升数字、capsule 分类和对象粒度默认都视为当前策略层，而不是不可变公理。

5. State Model

共享真相必须由事件来源和物化状态共同支撑

Authority of Truth: 系统中只能存在一个共享稳定状态面。planner、agent、adapter、memory compiler、external ecosystem 都只能提交候选更新，不能直接宣布真相成立。

Truth Shape

工程上更可取的实现是 append-only event log + materialized state view。共享稳定状态面不是一份任意可改的大对象图，而是从事件来源持续物化出的当前视图；receipt、grant、verification、memory 和 capsule 都必须能反查到其事件来源。

Fact Time and Conflict Semantics

Fact 至少要有来源、观测时间、验证时间、TTL 或有效期、置信或证明级别、版本号、supersedes 和 conflict_set。异步外部变化、跨表面回读不一致、以及旧 verified fact 过期时，不允许静默覆盖；必须进入降级、并存或显式裁决。

6. Risk and Grant Model

Composite Risk Rule: R0-R3 只是治理带宽，不是风险本体。真实风险必须被计算为 action × target × context × sequence 的组合函数；同一动作在不同时间窗、不同租户、不同系统和不同累计序列里，风险可以完全不同。

Prepare-to-Commit Barrier: 任何从 Prepare 进入 Commit 的动作，都必须重新经过风险确认、授权生成、postcondition 定义、compensation 或 recovery 路径确认，以及显式 approval 绑定。

选择器的任务是选最稳、最便宜、最少打扰用户的路径

Disallow

Planner Boundary: planner 只能产出候选动作、候选分解、候选事实和候选 capsule，不能直接决定 R3 最终提交动作、风险边界、授权范围或 verified fact。attention_budget 必须显式进入 selector，避免系统因过度 ask 而把正确设计表现成更慢流程。

9. Verification and Recovery

验证器决定是否真的达成目标，恢复器决定失败如何收敛

Verification Method Order
  • 直接状态回读
  • 结构化结果校验
  • 跨来源一致性校验
  • 工件级证据校验
  • 外部系统确认
  • 人工确认
Failure and Recovery
  • 失败分类：policy_denied / planning_insufficient / execution_failed / verification_failed / resource_unavailable / human_abort
  • 恢复动作：retry / replan / rollback / compensate / escalate
  • rollback 是精确逆操作；compensate 是不可完全回滚时的补偿语义。

Compensation Contract: 对每个 Commit 和高外部性 Operate，adapter 都应声明：是否可逆、如何补偿、最大损失窗、最终一致性窗口、哪些副作用不可恢复。recovery 不是理念词，而是执行语义的一部分。

Terminal Rule: 只有验证器或人工决策可以推进 completed / failed / rolled_back / paused_for_human / cancelled。不存在“执行成功即完成”的旁路。

12. Multi-Agent Coordination

多 agent 是条件性优化器，不是默认工作模式

Spawn Conditions
  • 子任务天然可并行
  • 写冲突低
  • 需要不同专长或信息源
  • 合并结果可以通过统一证据模型裁决
  • 总收益高于协调成本
Coordination Primitives
  • Claim / Yield / Block / Request / Handoff / Merge
  • agent 只能认领节点、读状态、申请授权、提交候选结果和推进其持有节点状态。
  • agent 不得直接写 verified facts、绕过授权执行高风险动作或维护私有长期真相源。

Compatible

Not Compatible

14. Acceptance Gates

没有通过这些门槛，就不允许进入正式实施

Design Freeze

必须明确分层：不可变公理、默认协议、可调策略三层不能混写；Fact 时间/冲突语义、组合风险、selector 评分、compensation 语义和 capsule 验证语义必须冻结。

Implementation Entry

所有 Commit 类动作已显式切出，高风险动作已有确认、验证和 compensation 或 recovery 路径；共享真相面必须可追溯到事件来源，不依赖特定厂商或协议名称成立。

MVP

首个可用版本不要求全覆盖。默认先证明两条窄链路：Observe/Synthesize/Prepare 和有限 Operate/受控 Commit；单 agent、单状态面、单验证器、单恢复器即可。

Priority Rules

安全优先于成本；验证优先于速度；当前证据优先于历史记忆；人工显式确认优先于 procedure 命中；共享状态优先于 capsule；任务边界优先于生态便利；恢复优先于继续推进；提交边界优先于自动化流畅性。

Disqualifiers

若仍然把某协议或某厂商写成架构真理、仍然模糊 Prepare 和 Commit 边界、仍然允许外部生态绕过授权或验证，则设计视为未完成。

verified -> stale | revoked
stale -> candidate | verified | revoked
revoked -> terminal

ProcedureAsset:
draft -> trial | deprecated
trial -> trusted | deprecated
trusted -> deprecated
deprecated -> trial

事实会变旧，procedure 会退化。长期运行里不存在“永久有效”的 verified fact 或 trusted procedure。

Grant and Lease Statuses 
CapabilityGrant:
issued -> active | revoked | expired
active -> exhausted | revoked | expired
exhausted -> terminal
expired -> terminal
revoked -> terminal

Node Ownership Lease:
offered -> active -> expired | yielded | revoked

Resource Write Lease:
requested -> active -> expired | revoked

任何 lease 或 grant 一旦过期、撤销或耗尽，执行器必须立即停止继续产生副作用。

Implementation Invariants
  • 一个 node 同时只能有一个 active owner lease。
  • 同一资源范围内只能有一个 active high-risk write lease。
  • 同一个 commit 动作必须绑定短时独占 lease 和明确 approval_ref。
  • 没有 verification result 的 receipt 不能推进任务终态。
  • 没有 receipt 的 action 不得进入 procedure 编译候选。
  • 物化状态面必须能够从事件来源重建，而不是只有当前快照。

  • 所有 required 字段缺失都必须在 schema 校验阶段被拒绝，而不是运行时默默补默认值。

Causality Rules
  • grant.issued 必须引用对应的 grant.request 或策略判定来源。
  • action.dispatched 必须引用 active grant 和目标 node。
  • receipt.recorded 必须引用 action_id 和 executor_id。
  • verification.recorded 必须引用 receipt_refs。
  • handoff.completed 必须引用 capsule_id。
  • memory.compiled 必须引用 verified evidence，而不是原始对话。
Canonical Flow Draft 
task.create
  -> task.created
  -> node.state_changed(created -> ready)
  -> grant.request
  -> grant.issued
  -> action.dispatch
  -> action.dispatched
  -> receipt.recorded
  -> verification.request
  -> verification.recorded
  -> node.state_changed(verifying -> terminal_or_next)

这条链路定义的是最小完整因果闭环。实现层可以批处理，但不能跳过 grant、receipt、verification 任一环节。

Recovery Flow Draft 
receipt.recorded(failed_or_partial)
  -> verification.recorded(verified_failure | insufficient_evidence | policy_violation_detected)
  -> recovery.request
  -> recovery.recorded(retry | replan | rollback | compensate | escalate)
  -> node.state_changed(...)

恢复不是异常旁路，而是正式状态推进链的一部分；尤其 rollback 也必须留下 receipt 和 verification 痕迹。

Handoff and Multi-Agent Flow 
lease.claim
  -> lease.issued
  -> work_in_progress
  -> capsule.generate
  -> capsule.published
  -> handoff.request
  -> handoff.completed
  -> lease.yield | new lease.issued

没有 capsule 的 handoff 只算产物转交，不算节点级正式交接。

Idempotency and Dedup Rules
  • 所有 command 和 event 都必须有稳定 id，用于去重和追踪。
  • 消费者必须能按 event_id 去重，不能假设消息只投递一次。
  • 同一 action 的重复派发必须由 idempotency_key 或 grant/lease 约束收敛。
  • 同一状态转换不能被两个独立事件重复记账为两个成功事实。
Machine-Checkable Interaction Invariants
  • 没有 active grant，不得出现 action.dispatched。
  • 没有 receipt.recorded，不得出现 verification.recorded 的 success/failure 结论。
  • 没有 verification.recorded，不得出现终态级 node.state_changed。
  • 没有 capsule.published，不得出现正式 handoff.completed。
  • 没有 verified evidence，不得出现 memory.compiled 进入 trusted 或 verified 层。
Non-Goals of This Appendix
  • 不规定事件总线必须中心化还是嵌入式。
  • 不规定 command 和 event 是否共用同一物理 topic 或日志。
  • 不规定同步 RPC 和异步事件谁先落地。
  • 不规定 tracing、metrics、audit 是否由同一系统承载。
  • 不规定重放和归档的存储格式。

Replay 1 · Research Brief

链路是 Observe -> Synthesize -> Prepare。它验证的是：候选 fact 不得越过证据直接进入 verified；Prepare 产物可以完成，但不等于 Commit；冲突 fact 可以并存，不得静默覆盖。

本回放里真正不可变的是真相独立于上下文、事实要带时间和来源、终态必须经验证；可调的是任务分类语言、事实 freshness 窗口、以及 draft 是否默认 auto-complete。

Replay 2 · CRM Update and Follow-up Commit

链路是 Observe -> Prepare -> Operate -> Commit。它验证的是：Prepare/Commit 屏障不能被绕过；R2/R3 风险要结合目标对象和序列环境计算；可逆写入和不可逆外发必须分别走 rollback 与 compensate 语义。

本回放里真正不可变的是 grant、verification、compensation 语义和事件来源；可调的是风险阈值、approval UX、selector 打分权重和补偿时窗。

Invariant Surfaces

单一真相源、事件可追溯、显式授权、验证后终态、不可逆动作的 compensation 语义。

Policy Surfaces

任务语言、风险阈值数字、路径偏好顺序、procedure 晋升门槛、capsule 形态、attention budget 触发阈值。

MVP Guidance

这两条 replay 就是建议的首个窄 MVP：先用单 agent、单状态面、单验证器把它们跑通，再扩多 agent 和更宽生态兼容。

Seal Clauses

6 条补充封口条文

F1

系统中只能存在一个共享稳定状态面作为唯一真相源，任何外部来源都只能提交候选更新。

F2

所有 verified fact 都是条件性的；时效过期、环境变化、冲突证据或用户撤销都会触发降级复核。

F3

任何从 Prepare 进入 Commit 的动作都必须重新过风险、授权、postcondition 和 recovery 检查。

F4

trusted procedure 永不自动放行 R3，高风险控制权始终留在授权和验证链路。

F5

Monitor 可以自动触发 Observe、Synthesize、Prepare 和有限 Operate，但绝不自动触发 Commit。

F6

外部能力、workflow、memory、agent 输出最高只到候选层，不能直接生成 verified fact、completed task 或 trusted procedure。
