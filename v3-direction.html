April 8, 2026 · v3 direction freeze

不是再造一个全家桶 agent，而是做一个最小设备控制内核。

v3 的核心结论只有一句： 标准优先，主权内核最小，能力按包安装。 这套系统不再把模型、浏览器、VM、本地推理、插件和工作流都打成一个笨重参考实现，而是把真正需要自研的部分收敛成 broker、ticket、verifier、state machine、scheduler、sessionlog。 其余能力尽量建立在现有成熟生态和统一协议之上。

Kernel: broker + ticket + verifier + state machine
Interop: MCP first-class, skills as pack layer
Models: pluggable providers, not built-in runtimes
Packaging: core first, packs optional

本次冻结的核心 固定什么必须自己做，什么必须复用现有生态，什么只能通过统一契约接入。
主设计轴 自动化执行稳定性、长期效率、便捷集成、轻量发行，而不是把所有能力都内置。
默认姿势 先装 core，再按需安装 browser pack、platform pack、model provider pack、VM pack。
Delta From v2

v2 到 v3 的根本变化

v2 的问题不是方向错，而是把“参考实现”、“发行组合”和“架构边界”写成了一件事。v3 把这三件事彻底分开。

v2 tendency

把默认实现写得过重

  • 默认云后端和本地推理层写得过于靠前。
  • 浏览器、VM、本地模型容易被理解成基础安装的一部分。
  • Plugin、Skill、MCP 被写成扩展点，而不是标准互操作层。
  • 参考实现容易被误读为唯一正确的发行方式。
v3 correction

把系统收敛成最小内核 + 标准接入

  • 固定最小内核，只承载主权、调度、审计和验证。
  • MCP 作为当前优先兼容通道之一，用于工具与资源接入。
  • skills 降到工作流打包层，不再承担权限边界。
  • 模型后端全部改成 provider adapter，不默认内嵌运行时。
Principles

v3 设计原则

所有后续实现和取舍都以这五条原则为准，而不是以某个 demo 是否好看为准。

01

主权边界只留在内核

模型、skill、MCP server、browser worker、插件都不能直接拿宿主主权，只能提出受限请求。

02

统一协议优先于私有发明

能用 MCP 解决的工具与资源接入，不再发明第二套工具协议；能复用现有 skill bundle 的，不做封闭私有格式。

03

控制平面最小化

核心进程只保留状态机、票据、验证、审计和调度，不把浏览器、模型运行时、VM 资产塞进基础安装包。

04

执行优先级不变

仍然坚持 API / UIA / DOM / browser / pixels 的表面顺序，以及 cache / macro / deterministic / planner 的成本顺序。

05

复用生态，自己守边界

真正值得自研的只有 broker、ticket、verifier、scheduler、rollback，不是模型服务轮子，不是新 skills 标准。

Kernel Boundary

v3 内核到底负责什么

如果某个组件不改变主权边界、执行因果或验证闭环，它默认不属于内核。

Kernel owns 
core/
  state_machine
  scheduler
  policy_engine
  capability_broker
  ticket_signer
  verifier
  rollback_coordinator
  sessionlog
  artifact_index
  diagnostics
  task_api
必须自研 EventEnvelope 进入方式、任务生命周期、审批点、票据签发、postcondition 验证、失败收敛。
不应归内核 模型推理运行时、浏览器资产、VM 镜像、私有 skill 标准、额外工具协议。
平台特定内容 UIA / AX / AT-SPI、shell、窗口系统、隔离方式，全部落到 adapter 或 pack 层。
结果 内核安装包可以保持很小，启动和常驻都不会被可选大组件拖累。
Interop Strategy

标准与成熟生态的接入策略

v3 不忽略现有生态，而是把它们放在明确位置上：MCP 是互联主协议，skills 是打包层，模型接口是 provider 层。

Domain v3 stance Why
MCP 当前优先兼容对象。更倾向由互操作层实现 MCP host，管理 tools、resources、prompts 的接入与生命周期。 MCP 已经是现成统一协议，适合作为当前首选兼容入口之一，不必再造第二套同类协议。
Skills 工作流/知识包层。支持导入导出，允许 adapter 兼容主流 bundle 结构。 skills 适合封装模板、流程、说明和资源，但不应该承担权限边界。
Model providers 全部走 provider adapter。云端、本地、远程、自托管接口都视为可替换后端。 模型后端变化太快，不应写死到核心架构里。
Native OS automation 通过 adapter 暴露为统一 capability，而不是直接让外部组件绕过 broker 调主机。 桌面语义层和宿主控制属于设备代理真正差异化能力。
MCP

协议解决互联

任何来自 MCP 的工具调用都要先映射成内核可理解的 capability scope，再由 broker 判定能否执行。

Skills

打包解决复用

skill 负责描述“怎么做”，不直接获得“能不能做”。skills 可以引用 MCP、adapter 和 policy template。

Providers

后端解决推理

planner、classifier、summarizer、reranker 只定义抽象接口，具体由任意兼容 provider 实现。

Four-Layer Stack

v3 四层结构

v3 不是三层模型代理图，而是四层产品结构图：内核、互操作层、原生适配层、能力包。

Layer 0

Kernel

Broker、ticket、policy、scheduler、verifier、rollback、sessionlog、diagnostics。

Layer 1

Interop

MCP host、skill pack adapter、model provider adapter。负责接入现有生态，不拥有宿主主权。

Layer 2

Native Adapters

UIA / DOM / browser / pixel / filesystem / process / window / clipboard / notification。

Layer 3

Capability Packs

browser pack、platform pack、VM pack、provider pack、workflow pack。全部按需安装。

Key consequence

这四层的因果关系必须保持单向：生态能力可以接进来，平台能力可以暴露出来，但任何外部能力都不能绕过内核直接获得主权。

Stable Contracts

统一契约先定死，再谈具体实现

v3 的复用能力不来自“大家都写一个插件”，而来自“大家都说同一种任务语言”。

EventEnvelope

规范化事件输入。来源、时间、上下文摘要、风险提示、可用表面。

CapabilityScope

动作类别的统一命名空间，例如 filesystem、browser、window、message、credential。

CapabilityTicket

步骤级授权载体，绑定 scope、resource、TTL、approval_ref、postcondition。

ExecutionStep

内核下发的单步动作定义，明确 adapter、输入、预算、预期工件和完成条件。

ExecutionReceipt

执行器返回的已完成证据，包含状态、工件引用、环境摘要、重试信息。

VerifierResult

postcondition 检查结果，决定 completed、failed、rolled_back 或 paused_for_human。

Interop rule 
external capability -> adapter/provider/skill/MCP entry
                  -> normalize into EventEnvelope / ExecutionStep
                  -> broker evaluates policy and issues CapabilityTicket
                  -> executor runs
                  -> verifier decides result

No direct host action is allowed outside this path.
Capability Packs

能力包不是附属品，而是发行策略核心

v3 默认只交付最小内核。浏览器、平台适配、隔离执行和推理后端全部改成可选 pack。

Core Pack

必须安装

内核、状态存储、票据、验证、诊断、本地控制台、基础 task API。

Platform Pack

按 OS 安装

Windows UIA、macOS AX、Linux AT-SPI，以及各平台 shell、窗口、通知、剪贴板接口。

Browser Pack

按需安装

DOM adapter、browser worker、isolated profile 管理、trace 与 artifact 回收。

Provider Pack

按后端安装

planner、classifier、summarizer 等 provider adapter，可接云模型、远程接口或本地服务。

Workflow Pack

按场景安装

skill bundles、policy templates、macro packs、recovery recipes，不改内核代码。

VM Pack

重能力，默认不装

高风险隔离执行、镜像/快照、桥接票据、工件回收。只为真正需要隔离执行的部署提供。

发行原则 基础安装不应包含浏览器资产、本地模型运行时和 VM 镜像。
能力发现 未安装的 pack 必须在 diagnostics 和 capability registry 里明确显示为 unavailable，而不是静默失败。
复用目标 第三方可以只实现一个 pack 或 adapter，就接入整套执行内核。
Execution Model

自动化执行路径在 v3 里怎么走

v3 改的是产品结构，不改执行本质。执行仍然服从低成本优先、结构化优先、结果验证优先。

v3 scheduling policy 
event arrives
  -> normalize
  -> load relevant policy / semantic / procedural memory
  -> cache hit ? use cached plan
  -> macro hit ? use workflow macro
  -> deterministic adapter available ? use direct/script path
  -> MCP resource/tool sufficient ? map to capability and execute
  -> planner needed ? call provider adapter
  -> broker signs ticket
  -> executor runs
  -> verifier checks postcondition
  -> update memory / macro / sessionlog
便捷性优势 MCP、skills、native adapters 都走统一入口，用户不需要自己手动切换多套 agent 工作流。
自动化优势 不再依赖某个模型临场把所有步骤想出来，重复流程会沉淀成宏，执行越来越稳定。
效率优势 planner 只在真正需要时被唤醒，cache 和 macro 命中后几乎不再消耗 frontier token。
工程优势 第三方只需要遵守统一契约，就可以复用整个执行内核和诊断体系。
Non-Goals

v3 明确不做什么

这些取舍不是能力不足，而是为了避免系统重新膨胀回一个难以发布、难以复用的大杂烩。

Do not do this
  • 不发明新的工具协议替代 MCP。
  • 不发明新的封闭 skills 标准要求生态重写。
  • 不把某一家模型 API 写成架构中心。
  • 不默认自带本地模型推理运行时。
  • 不把浏览器资产、模型资产、VM 镜像都塞进基础安装包。
What remains custom
  • 主权内核：broker、ticket、verifier、scheduler、rollback、sessionlog。
  • 宿主适配：桌面语义层、窗口系统、文件/进程/通知等设备控制。
  • 统一契约：事件、步骤、票据、工件、验证结果。
  • 治理层：审批、审计、诊断、degraded mode、failure convergence。
Acceptance Gates

v3 成功的最低验收门槛

这份方向文档的目标不是更漂亮，而是后续实现不会再走回“大而全且难发布”的老路。

Boundary

内核边界清楚

核心模块必须能在不安装 browser pack、VM pack、provider pack 的情况下独立运行。

Interop

MCP 是默认能力入口

外部 tools/resources/prompts 的首选接入方式是 MCP，而不是私有插件协议。

Packaging

基础安装保持轻量

基础安装包不包含浏览器大资产、本地模型运行时和 VM 镜像。

Providers

模型后端可替换

planner、classifier、summarizer 不依赖单一厂商接口，也不假设本地模型必须内嵌。

Automation

确定性路径优先命中

固定任务集里，稳定任务应优先走 cache、macro、native adapter 或 MCP 工具，而不是 planner + pixel 兜底。

Economics

越用越便宜

宏命中率上升，frontier token、截图次数、人工切换成本和安装包默认体积都随时间下降。

v3 one-line summary

这不是一个更大的 agent 方案，而是一个更小的控制内核方案。 它的目标不是替代现有 MCP、skills、provider 生态，而是在这些生态之上提供一个统一、可验证、可发布、可长期演进的设备级执行平面。
